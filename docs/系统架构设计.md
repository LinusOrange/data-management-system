# 票据数据对账系统（Web 端）系统架构设计（V1）

## 1. 目标与范围

本系统用于处理来自**两个账目文件**的采购/票据信息，完成以下核心能力：

- 导入并解析两个来源文件（来源 A、来源 B）。
- 基于可配置规则执行对账，沉淀“按条采购记录”的对账状态。
- 按条维护“是否完成对账”“是否完成开票”等业务状态。
- 提供系统看板，实时展示：
  - 尚未对账条数
  - 尚未开票条数
  - 对账异常条数
  - 已完成闭环条数
- 支持后续扩展（你下一条 prompt 给出具体文件结构与对账规则后可快速落地）。

## 2. 总体架构

采用分层架构：**前端展示层 + 应用服务层 + 数据处理层 + 数据存储层**。

```text
[Web 前端]
  ├─ 看板 Dashboard
  ├─ 对账任务管理
  ├─ 采购明细列表/详情
  └─ 异常处理与人工复核
        │
        ▼
[后端 API 服务]
  ├─ 认证与权限（可后续加）
  ├─ 采购记录管理 API
  ├─ 对账任务 API
  ├─ 开票状态维护 API
  └─ 看板统计 API
        │
        ▼
[对账引擎与任务调度]
  ├─ 文件导入/解析器（A/B）
  ├─ 字段标准化
  ├─ 匹配与对账规则执行
  ├─ 差异检测与异常分类
  └─ 结果回写（状态、日志）
        │
        ▼
[数据库]
  ├─ 原始文件批次表
  ├─ 标准化明细表
  ├─ 对账结果表
  ├─ 开票状态表
  └─ 审计日志表
```

## 3. 核心业务对象（领域模型）

> 先定义稳定对象，后续你给具体字段时再细化。

1. **导入批次（ImportBatch）**
   - 标识一次文件上传与处理过程。
   - 包含来源类型（A/B）、上传人、上传时间、处理状态。

2. **采购记录（PurchaseRecord）**
   - 以“采购业务条目”为主实体。
   - 保存标准化后的关键字段（如订单号、供应商、金额、日期等）。
   - 业务状态：
     - `reconciliation_status`（未对账/已对账/异常）
     - `invoice_status`（未开票/已开票/部分开票）

3. **对账结果（ReconciliationResult）**
   - 记录 A/B 两侧匹配关系与差异明细。
   - 保存匹配置信度、差异类型、规则命中情况。

4. **异常工单（ExceptionCase，可选）**
   - 对无法自动闭环的记录形成异常待办。
   - 支持人工处理、备注、指派、关闭。

5. **操作审计（AuditLog）**
   - 记录状态变更历史，保证可追溯。

## 4. 关键流程设计

### 4.1 文件导入与入库流程

1. 用户上传来源 A / 来源 B 文件。
2. 系统创建导入批次，保存文件元信息。
3. 解析器读取文件并进行字段标准化。
4. 标准化数据写入“标准化明细表”。
5. 触发对账任务（异步）。

### 4.2 对账处理流程

1. 对账引擎读取待处理批次。
2. 按规则进行匹配（主键匹配 + 补充匹配策略）。
3. 输出结果：
   - 匹配成功 → 标记“已对账”
   - 存在差异 → 标记“对账异常”
   - 无匹配项 → 标记“未对账”
4. 回写 `PurchaseRecord` 和 `ReconciliationResult`。
5. 产出看板统计数据。

### 4.3 开票状态维护流程

1. 业务人员在列表或详情页更新开票状态。
2. 系统记录状态变更与操作人。
3. 如开票完成，触发闭环检查：
   - 已对账 + 已开票 => 计入闭环完成。

## 5. 数据库设计建议（V1）

建议 PostgreSQL（事务能力强、JSON 扩展友好）。

### 5.1 表结构草案

- `import_batch`
  - `id`, `source_type`, `file_name`, `uploaded_by`, `uploaded_at`, `status`, `error_message`

- `purchase_record`
  - `id`, `business_key`, `source_type`, `supplier_name`, `order_no`, `amount`, `biz_date`
  - `reconciliation_status`, `invoice_status`
  - `reconciliation_result_id`, `last_reconciled_at`
  - `created_at`, `updated_at`

- `reconciliation_result`
  - `id`, `left_record_id`, `right_record_id`, `match_status`, `diff_type`, `diff_detail_json`, `rule_version`, `created_at`

- `invoice_event`
  - `id`, `purchase_record_id`, `old_status`, `new_status`, `operator`, `remark`, `created_at`

- `audit_log`
  - `id`, `entity_type`, `entity_id`, `action`, `operator`, `before_json`, `after_json`, `created_at`

### 5.2 索引建议

- `purchase_record(business_key)`
- `purchase_record(reconciliation_status, invoice_status)`
- `purchase_record(order_no, supplier_name, amount)`
- `reconciliation_result(match_status, diff_type)`

## 6. API 设计草案

### 6.1 导入与任务

- `POST /api/imports`：上传文件并创建批次
- `GET /api/imports/{id}`：查询批次处理进度
- `POST /api/reconciliation/run`：手动触发对账（可选）

### 6.2 采购记录与状态维护

- `GET /api/purchases`：分页查询（支持按状态筛选）
- `GET /api/purchases/{id}`：查询单条详情及对账差异
- `PATCH /api/purchases/{id}/invoice-status`：更新开票状态
- `PATCH /api/purchases/{id}/reconciliation-status`：人工修正对账状态（需权限）

### 6.3 看板统计

- `GET /api/dashboard/summary`
  - 返回：
    - `pending_reconciliation_count`
    - `pending_invoice_count`
    - `reconciliation_exception_count`
    - `closed_loop_count`

## 7. 前端页面信息架构

1. **登录页（可后补）**
2. **看板页 Dashboard**
   - KPI 卡片：未对账、未开票、异常、闭环完成
   - 趋势图：按日新增异常、按日闭环量

4. **采购明细页**
   - 表格筛选：对账状态、开票状态、供应商、日期范围
   - 行内操作：更新开票状态、查看差异详情

5. **异常处理页**
   - 展示差异字段与建议处理动作
   - 支持人工确认/驳回/备注

## 8. 技术选型建议（可快速落地）

- 前端：React + Ant Design（列表/看板效率高）
- 后端：Node.js (NestJS) 或 Python (FastAPI)
- 数据库：PostgreSQL
- 异步任务：Redis + BullMQ / Celery
- 文件存储：本地（开发）→ 对象存储（生产）
- 可观测性：日志 + 任务执行指标 + 异常告警

---

## 9. 非功能性设计

- **一致性**：对账结果回写采用事务，防止部分成功。
- **可追溯**：状态变更必须写审计日志。
- **幂等性**：同一批次重复触发时避免重复写入。
- **可扩展**：规则引擎版本化，支持将来切换规则。
- **安全性**：文件上传校验、接口鉴权、操作留痕。
